The install block is out of date and will fail as written. The project moved off MichMich to the MagicMirrorOrg org, and MagicMirrorSquared isn't the canonical source. The repo is now at MagicMirrorOrg and the install command changed from npm install. Both npm install and node serveronly are old. The current manual flow is: Pidiylab
git clone https://github.com/MagicMirrorOrg/MagicMirror.git
cd MagicMirror
node --run install-mm        # not "npm install"
cp config/config.js.sample config/config.js
node --run server            # serveronly mode — not "node serveronly"
Server-only mode is now started with node --run server, and on a Pi 2 the install step runs around 25 minutes — you're in that ballpark on the Zero 2 W, so don't kill it when the terminal goes quiet. The other trap is Node itself: Pi OS's apt Node is too old, recent MagicMirror wants a current LTS, and the node --run syntax needs Node 22+. Rather than fight that manually, use sdetweil's community installer, which checks and installs the right Node version, clones the repo, and sets up pm2: MagicMirror + 2
bash -c "$(curl -sL https://raw.githubusercontent.com/sdetweil/MagicMirror_scripts/master/raspberry.sh)"
rc.local is the wrong tool for the browser half of autostart. Split the two jobs. The MM server should run under pm2 (the installer above wires this up; pm2 startup + pm2 save makes it survive reboot) — it gives you restart-on-crash and logs that rc.local won't. Chromium, though, can't launch from rc.local: there's no display session that early in boot, so it just fails. Put the kiosk command in the desktop session's autostart, and have it wait until port 8080 is actually listening — otherwise Chromium loads before the server is up and shows a connection error that --noerrdialogs won't paper over. One more wrinkle: current Raspberry Pi OS (Bookworm) defaults to Wayland rather than X11, so older X11 kiosk snippets may not apply cleanly — match your launch method to the session you're actually running. width: 800, height: 480 in config.js does nothing in serveronly mode. Those are Electron window options, and you have no Electron window — resolution comes from the Pi's display config plus Chromium going fullscreen under --kiosk. Drop it from config.js and set 800×480 at the OS/display level; --kiosk fills whatever the panel reports. Your address: "localhost" / port: 8080 is correct for a same-Pi kiosk; you'd only need address: "0.0.0.0" plus an ipWhitelist entry to reach it from another device, which is handy if you want to debug from your laptop during setup. The risk that's not on your list: 512MB of RAM. It bites in two places. At install time, compiling native deps on 512MB can OOM and die mid-build, so add swap or zram before you install (you can dial it back after). At runtime, the memory hog is Chromium, not node serveronly — so give it room: use Raspberry Pi OS Lite (no full desktop) with a minimal X/Wayland + Chromium rather than the desktop image, keep the GPU memory split modest, run zram, and keep modules few, which you're already planning. If it still thrashes, the documented fallback is the client/server split — but with the panel wired directly to the Zero 2 W that likely means rethinking the display, so try the lean-OS route first.
On the leftovers: PurpleAir still needs that free API key (read key, X-API-Key header against api.purpleair.com/v1/...), so check the 3rd-party module list for an existing air-quality module before writing your own — a ~50-line custom module is realistic if nothing fits. And for version control, don't track the whole ~/MagicMirror install; keep just config.js (secrets gitignored or pulled from process.env) and any custom MMM-* module in your repo, and symlink them into the install. Keeping the repo is the right call — archiving artifacts/ can wait until MM is actually up.
