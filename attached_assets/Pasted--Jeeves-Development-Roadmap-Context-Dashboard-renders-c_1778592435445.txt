# Jeeves — Development Roadmap

## Context
Dashboard renders correctly in browser with test data. Next steps are to get it physically running on the Pi, then progressively replace TEST_DATA with real integrations one tile at a time. Home automation hub is likely Home Assistant (TBD). Weather source and API server location are TBD.

---

## Phase 1 — Pi Setup (kiosk on hardware)
*Goal: dashboard running unattended on the screen, survives reboots.*

**Tasks:**
1. Copy `dashboard.html` onto the Pi (scp or git clone)
2. Install Chromium if not present: `sudo apt install chromium-browser`
3. Disable screen blanking / screensaver
4. Create autostart entry so Chromium launches in kiosk mode on boot:
   - `/etc/xdg/autostart/dashboard.desktop` or `~/.config/autostart/`
   - Command: `chromium-browser --kiosk --noerrdialogs --disable-infobars --disable-session-crashed-bubble file:///path/to/dashboard.html`
5. Verify at 800×480 on the physical display

**Files affected:** none in this repo (Pi config only)

**Done when:** Pi boots to dashboard, clock ticks, test data renders.

---

## Phase 2 — API Server skeleton
*Goal: replace TEST_DATA with a real HTTP endpoint so subsequent phases just add data sources.*

**Recommended approach:** Small Python server on the Pi (`server.py`) that:
- Exposes `GET /api/status` returning the same JSON shape as TEST_DATA
- Starts with hardcoded/stub values (identical to TEST_DATA)
- `dashboard.html` `fetchData()` swapped to `fetch('/api/status')`

Why Python on the Pi: lightweight, Home Assistant has a Python API, no extra runtime needed.

**Files to create:**
- `server.py` — Flask or http.server, single file, mirrors TEST_DATA shape initially
- `requirements.txt` — flask only (or none if using stdlib http.server)

**Files to modify:**
- `dashboard.html` — uncomment the real `fetch('/api/status')` in `fetchData()`, remove TEST_DATA

**Done when:** dashboard loads live from `localhost` instead of inline JS object.

---

## Phase 3 — Weather (first real data)
*Goal: real temp/conditions, easiest win with no auth.*

**Recommended source:** [Open-Meteo](https://open-meteo.com/) — free, no API key, no signup.
- Geocode location once (hardcode lat/lon)
- Call `api.open-meteo.com/v1/forecast` from the server (not the browser — avoids CORS)
- Map response fields to the `weather` shape

**Files to modify:**
- `server.py` — add weather fetch, cache result for ~10 min (Pi Zero W: avoid hammering external APIs)

**Done when:** weather tile shows real conditions and forecast.

---

## Phase 4 — Home Assistant integration (tiles, one at a time)
*Goal: progressively replace stub tile values with real sensor data.*

**Likely hub:** Home Assistant (confirm before starting).
HA exposes a REST API: `GET /api/states/<entity_id>` with a long-lived access token.

**Order of attack (easiest → hardest):**
1. **Thermostat** — HA `climate.*` entity, straightforward read
2. **Who's home** — HA `device_tracker.*` or `person.*` entities
3. **Front door / garage** — HA `binary_sensor.*` (contact sensors)
4. **Pool** — depends on sensor type (likely a temperature sensor entity)
5. **Laundry** — typically a power-monitoring smart plug watching wattage; HA automation fires when wattage drops → set `alert: true`

For each tile:
- Add HA entity ID to a config block at the top of `server.py`
- Fetch state in the `/api/status` handler
- Map HA state string → tile `value`, `alert`, `degraded` fields

**Files to modify:** `server.py` only (one tile per PR/commit, easy to test and roll back)

**Done when:** all 6 tiles show live data; `alert: true` fires correctly for laundry.

---

## Phase 5 — Hardening
*Goal: dashboard survives network blips, Pi reboots, HA restarts.*

- `server.py`: per-source try/except → set `degraded: true` on that tile if fetch fails (rest of dashboard keeps working)
- Add a systemd service for `server.py` so it restarts on crash
- Consider a watchdog script that checks if Chromium is running and relaunches if not
